2026-07-22 — delete a contract from the register
- Deleting a contract is not the same as ending one, so the rule lives in
  domain/contractDelete.ts (canDeleteContract) rather than in a button handler:
  * HARD BLOCK when any RAR is billed against the contract. RARs record work
    measured and money paid; removing their contract would leave payments
    attached to something that no longer exists — a hole in the audit trail, not
    a correction. The message names the blocking RARs and points at the remedy
    (remove the RARs first, or close the contract instead). Blocks equally for a
    draft RAR and a paid one.
  * WARN, do not block, otherwise. An awarded/in-progress contract records a
    commitment actually made, and its approval chain goes with it; locked BOQ
    lines are released back to the planner. But a contract awarded to the wrong
    subcontractor is a real mistake needing a real way out — refusing outright
    would leave a wrong record in the register permanently, which is worse than a
    recorded deletion.
- provider.deleteContract(projectId, contractId) re-checks the rule ITSELF, so a
  caller that skipped the dialog still cannot orphan payment records, and writes
  an audit entry naming the contract, value, status and how many BOQ lines were
  released. Api provider stubs DELETE /contracts/:id.
- UI: a Delete action per row opens a confirmation showing either the block
  reason or the consequences. When blocked, the confirm button is not rendered at
  all — there is no way to force it through. Added .btn-danger styling.
- Locked quantities release automatically: itemLocks derives from the contracts
  list, so removing the contract frees its lines with no extra bookkeeping.

2026-07-23 (b) — BOQ item identity was POSITIONAL (silent re-pointing hazard)
- Found while answering "how do I choose the corrected BOQ?". The advice would
  have been "add the Toll Plaza to the project BOQ and re-import" — and that
  advice would have corrupted the contract.
- replaceBoq() minted ids as `boq-<project>-<row index>`. Contract lines, RAR
  lines, allocations and BOQ↔WBS mappings all store boqItemId. So inserting ONE
  row (a provisional sum in bill order, say) shifted every index below it and
  silently re-pointed every stored reference at a DIFFERENT item. No error, no
  warning — the contract would simply start describing the wrong work.
- FIX: identity is now natural, not positional. An incoming row matching an
  existing item on (bill, code, description) keeps that item id; only genuinely
  new rows get new ids, numbered beyond every id already in use so a retired
  index is never recycled onto a different item. Duplicate natural keys are
  consumed one-for-one (shift) so two identical rows take two distinct items.
- Tests data/boqIdentity.test.ts (5): ids survive a mid-list insertion; a CONTRACT
  LINE still points at the same item after a provisional sum is inserted above it
  (the exact corruption); qty/rate update in place without an id change; two
  identical rows get two ids; a removed item id is never recycled.
- NOTE still open: BoqImport gives no warning when replacing a BOQ whose items are
  referenced by contracts — a row REMOVED on re-import orphans its contract lines.
  Identity is now safe; the orphan warning is not built yet.
